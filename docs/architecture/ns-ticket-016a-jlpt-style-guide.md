# NS-TICKET-016A: JLPT형 문항 스타일 가이드

## 목적

기존 JLPT 쇼츠 데이터(`word`, `hiragana`, `meaning`, `level`, `score`, `used`, `day`)를 단순 뜻 맞히기 문제가 아니라 **JLPT 학습형 연습문항**으로 변환하기 위한 생성 규칙이다.

이 문서는 공식 JLPT 샘플 페이지 `https://www.jlpt.jp/samples/sample09.html`의 N1~N5 문제 예시 PDF를 **문항 구조와 레이아웃 패턴만 분석**해 만든 내부 스타일 가이드다. 공식 문제 문장/선택지/지문은 복제하지 않는다.

중요: 공식 샘플 PDF는 **형식 참고 자료**일 뿐이다. 문법/어휘/독해 문항을 만들 때 PDF 속 문제를 변형, 패러프레이즈, 일부 치환, 난이도 조정, 지문 축약 방식으로 재사용하지 않는다. 생성 문항의 실제 내용은 쇼츠 데이터와 자체 작성 문장으로만 만든다.

청해(`聴解`)는 sample09에 MP3와 script가 함께 제공되지만, 현재 JLPT Quiz MVP에서는 생성하지 않는다. 청해는 별도 audio/player/schema가 필요하므로 후속 티켓으로 분리한다.

## Trust 원칙

금지 표현:

- 공식성·기출성을 암시하는 표현
- 실제 시험 출제를 예측하거나 보장하는 표현
- 출제 확률을 단정하는 표현
- 공식 출제 경향을 확정적으로 말하는 표현

허용 표현:

- JLPT 학습형 연습문항
- JLPT 문항 형식에 가까운 연습문제
- 공식 샘플의 형식적 구조를 참고한 자체 제작 문제
- 사용자 제보 기반 출제 체감 score

## 공통 문항 규칙

- 최우선 품질 기준은 `일본어 문장이 자연스러운가`와 `JLPT 문항 형식에 가까운가`이다.
- 위 두 기준을 통과하지 못하면 선택지 품질이나 해설이 좋아도 import하지 않는다.
- 모든 문항은 보기 4개를 사용한다.
- 정답은 `A/B/C/D`로 저장하되, 화면에는 제출 전 정답/해설을 노출하지 않는다.
- 문제 본문은 일본어로 작성한다.
- 보기 4개는 일본어로 작성한다. 한국어 뜻풀이 보기는 사용하지 않는다.
- 한국어는 답안 제출 후 앱 해설 또는 관리 검수용 설명에만 사용한다.
- 문제 풀이 전 화면에는 한국어 문제문, 한국어 해설, 정답 의미, 정답 선택지를 노출하지 않는다.
- “다음 문법 표현의 의미로 가장 알맞은 것은?”처럼 한국어로 의미를 묻는 문항은 JLPT형 문항으로 보지 않고 폐기한다.
- 보기 4개는 품사/활용형/길이가 최대한 비슷해야 한다.
- 오답은 터무니없는 단어가 아니라 학습자가 헷갈릴 만한 표현으로 만든다.
- 같은 문항 안에서 보기 텍스트가 중복되면 안 된다.
- 저작권 보호를 위해 공식 샘플의 문장, 지문, 보기 조합을 그대로 재사용하지 않는다.
- 공식 샘플의 문장, 지문, 보기 조합을 변형하거나 패러프레이즈해서도 안 된다.
- sample09의 PDF/MP3/script는 레포에 저장하지 않는다. 필요한 경우 임시로 확인한 뒤 삭제하고, 문서에는 추상화한 형식명만 남긴다.

## Vocab 문항 유형

### V1. 한자 읽기 선택형

공식 샘플에서 확인한 구조:

- 문장 안의 한자어 또는 단어에 밑줄이 있다.
- 질문은 “밑줄 친 말의 읽기로 가장 알맞은 것”을 고르는 형식이다.
- 보기 4개는 모두 히라가나 읽기다.

생성 규칙:

- `word`가 한자 포함 단어일 때 우선 사용한다.
- `question_text`에는 자연스러운 일본어 문장과 밑줄 표시 대상 단어를 넣는다.
- 보기에는 정답 `hiragana`와 발음이 비슷한 오답 3개를 넣는다.
- 한자 없는 히라가나 단어는 이 유형에서 제외한다.

DB 매핑:

```text
question_type: vocab_reading
question_text: 문장 + 읽기 대상 단어 표시
choice_a~d: 히라가나 보기
correct_choice: 정답 보기
explanation: 대상 단어의 읽기와 의미 설명
```

### V2. 문맥 빈칸 어휘 선택형

공식 샘플에서 확인한 구조:

- 한 문장 안에 `（　　　）` 빈칸이 있다.
- 빈칸에 들어갈 가장 자연스러운 어휘를 보기 4개 중 고른다.
- 보기는 같은 품사 또는 비슷한 문장 기능을 가진 단어들이다.

생성 규칙:

- 쇼츠 데이터의 `word`를 정답 후보로 사용한다.
- 정답 단어가 자연스럽게 들어가는 짧은 일본어 문장을 새로 만든다.
- 오답은 같은 JLPT 레벨 또는 인접 레벨의 같은 품사 단어를 사용한다.
- 문장만 보고 정답이 너무 쉽게 드러나지 않게 하되, 문맥상 하나만 자연스럽게 만든다.

DB 매핑:

```text
question_type: vocab_context_blank
question_text: 빈칸 포함 일본어 문장
choice_a~d: 어휘 보기
correct_choice: 정답 보기
explanation: 왜 정답 어휘가 문맥상 자연스러운지 설명
```

### V3. 의미 유사/바꿔 말하기 선택형

공식 샘플에서 확인한 구조:

- 문장 안의 밑줄 친 표현과 의미가 가장 가까운 것을 고른다.
- 보기는 한국어 번역이 아니라 일본어 유의 표현/설명이다.

생성 규칙:

- N3 이상에서 우선 사용한다.
- 정답은 단순 번역이 아니라 일본어 유의어 또는 짧은 일본어 설명으로 만든다.
- 오답은 의미 영역은 가깝지만 문맥상 맞지 않게 만든다.

## Grammar 문항 유형

### G1. 문장 빈칸 문법 선택형

공식 샘플에서 확인한 구조:

- 문장 일부가 `（　　　）` 또는 빈칸으로 비어 있다.
- 보기 4개 중 문법적으로 가장 자연스러운 표현을 고른다.
- 보기는 활용/접속/의미가 비슷한 문법 표현이다.

생성 규칙:

- 공식 PDF에 나온 문법 문제를 변형하지 않는다. 빈칸 위치, 앞뒤 문맥, 보기 조합이 PDF와 닮아 있으면 폐기한다.
- `word` 또는 `grammar_point`를 정답 문법으로 사용한다.
- 문법 표현의 접속 조건에 맞는 앞말/뒷말을 반드시 구성한다.
- 오답은 형태가 비슷하지만 접속 또는 의미가 어긋나는 표현으로 만든다.

DB 매핑:

```text
question_type: grammar_sentence_blank
question_text: 빈칸 포함 일본어 문장
choice_a~d: 문법 표현 보기
correct_choice: 정답 보기
explanation: 접속/의미/문맥 기준 설명
```

### G2. 별표 어순 배열형

공식 샘플에서 확인한 구조:

- 문장 안에 여러 빈칸이 있고 그중 하나에 `★` 표시가 있다.
- 보기 4개는 문장 조각이다.
- 올바른 문장 순서를 만든 뒤, `★` 위치에 들어가는 번호를 답한다.

생성 규칙:

- N3 이상 또는 문법 구조가 충분히 긴 항목에서 사용한다.
- 정답 문장을 먼저 만든 뒤 4개 조각으로 분해한다.
- 조각은 조사/수식어/동사구가 섞이도록 하되, 해답이 하나로 수렴해야 한다.
- 앱 MVP에서는 `question_text`에 문장과 `★` 위치를 표시하고, `choice_a~d`에 조각을 넣는다.
- `correct_choice`는 `★` 위치에 들어가는 조각의 보기 키다.

주의:

- 이 유형은 UI 설명이 필요하므로 1차 import에서는 preview만 생성하고, 본격 적용은 별도 티켓에서 진행한다.

### G3. 연결 지문 문법 클로즈형

공식 샘플에서 확인한 구조:

- 짧은 글 하나에 여러 번호 빈칸이 들어간다.
- 각 번호마다 보기 4개 중 하나를 고른다.
- 문장 하나가 아니라 글 전체 흐름과 접속 관계를 본다.

생성 규칙:

- 현재 MVP의 단일 문항 DB에는 바로 넣기 어렵다.
- `passage_id`, `blank_no`, `passage_text` 같은 확장 schema가 필요하다.
- 1차 범위에서는 스타일 가이드에만 보관하고 실제 import는 보류한다.

## Reading 문항 유형

공식 샘플에서 확인한 구조:

- 짧은 글/공지/에세이/설명문을 읽고 질문에 답한다.
- 보기 4개 중 글쓴이 의도, 핵심 내용, 세부 정보, 지시 대상에 맞는 답을 고른다.
- 일부 지문에는 주석이 붙는다.

생성 규칙:

- 현재 vocab/grammar MVP와는 별도 `reading_passages`, `reading_questions` schema가 필요하다.
- 1차 범위에서는 import하지 않고, Phase 3 이후 확장으로 둔다.

## 1차 생성 우선순위

NS-TICKET-016B에서는 아래 두 유형만 생성한다.

```text
Vocab: V2 문맥 빈칸 어휘 선택형
Grammar: G1 문장 빈칸 문법 선택형
```

이유:

- 기존 DB 필드에 바로 매핑 가능하다.
- 공식 샘플의 구조적 느낌은 살리되, 저작권/복제 위험이 낮다.
- 사용자가 눈으로 품질 검수하기 쉽다.

## 생성 입력 포맷

```json
{
  "item_type": "vocab",
  "jlpt_level": "N5",
  "word": "会う",
  "reading": "あう",
  "meaning": "만나다, 보다",
  "source_day": "DAY 004",
  "source_used": true
}
```

## 생성 출력 포맷

```json
{
  "item_type": "vocab",
  "jlpt_level": "N5",
  "question_type": "vocab_context_blank",
  "question_text": "새로 작성한 일본어 빈칸 문장",
  "choice_a": "보기 A",
  "choice_b": "보기 B",
  "choice_c": "보기 C",
  "choice_d": "보기 D",
  "correct_choice": "A",
  "explanation": "정답이 자연스러운 이유와 한국어 의미 설명",
  "source_item": "원본 쇼츠 데이터 항목",
  "review_status": "draft"
}
```

## 사람 검수 체크리스트

- [ ] 일본어 문장이 자연스럽다.
- [ ] JLPT 문항 형식에 가까운 연습문제로 느껴진다.
- [ ] 공식 샘플 문장/보기/지문을 그대로 복제하지 않았다.
- [ ] 보기 4개가 중복되지 않는다.
- [ ] 정답이 하나로 수렴한다.
- [ ] 오답이 너무 터무니없지 않다.
- [ ] 문장이 해당 JLPT 레벨에 비해 과하게 어렵지 않다.
- [ ] 한국어 해설은 답안 제출 후에만 노출된다.
- [ ] 한국어 해설이 공식성 또는 출제 예측처럼 보이지 않는다.
- [ ] `unknown` feedback을 `no`로 합치지 않는 기존 score 정책과 충돌하지 않는다.
